Introduction

Selenium is simply a tool used to automate web browsers. It is primarily used as an automated User Interface tester, but it can be used for any browser automation task you want! (For example, automatically entering into an daily online contest.) In this tutorial we will focus on using Selenium as a tool to automate UI testing for web applications.

Why Use Selenium?

It is important for all layers of an application to be tested, but unfortunately testing the UI is sometimes forgotten. With Selenium, UI testing can be easily automated in a repeatable way that is suitable for regression testing. Selenium is also easily customizable to all your testing needs because it can be used in multiple languages and testing platforms, and it can run in multiple browsers and operating systems. Supported languages include Java, Python, C#, Ruby, and Perl. Supported browsers include Firefox, Internet Explorer, Chrome, Opera, iOS Driver, and Android Driver. Selenium should not be used, however, if the UI of a web application frequently changes or will change soon in the future. If the UI changes, all of the tests might have to change as well, which defeats the purpose of automating the tests.

Who Uses Selenium?

Anyone who wants to automate their work in a browser can use Selenium! It is commonly used by quality assurance (QA) engineers or other similar application testers.

How Does Selenium Work?

When you run a Selenium test, Selenium pulls up a browser window and triggers real events, such as clicking on a button, that could occur in your application. Therefore, many tests are written as a chain of events that would likely occur if a real user was using your application. The architecture is also quite simple. The Selenium test written in your specified language sits at the bottom, Selenium WebDriver runs on top of the test and below your specified web browser, and finally the web application sits at the top.

What is the Difference Between Selenium WebDriver, RC and IDE?

Selenium has multiple parts, all of which provide different advantages to you as a tester. For this tutorial we will only cover Selenium WebDriver. Selenium WebDriver allows you to create large suites of tests that can be automated and used for regression testing. It is described as a “collection of language specific bindings to drive a browser” by the official Selenium documentation, which just means that you can download Selenium for your preferred language to create test files that use Selenium capabilities. Selenium Remote Control is the ancestor of WebDriver, and is now officially deprecated. Selenium IDE is a lightweight version of WebDriver that allows a tester to write quick scripts for basic testing or for reproducing a bug. It is a simple Firefox add-on that records all interactions in the browser.

Commands and Operations

From here, we will run through an example Python script to demonstrate some of the capabilities of Selenium. At a high level, the example will navigate to IBM’s Wikipedia page in Firefox and perform some tests there. The full code can be seen below: 

from selenium import webdriver                          # import Python libraries
from selenium.webdriver.common.keys import Keys

driver = webdriver.Firefox()                            # open webdriver for Firefox

driver.implicitly_wait(10)                              # set max wait period for driver

driver.get("https://www.wikipedia.org")                 # navigate to Wikipedia

elem = driver.find_element_by_id("searchInput")         # find the search box
elem.clear()                                            # clear the search box

elem.send_keys("IBM")                                   # input "IBM" into the search box

elem.send_keys(Keys.RETURN)                             # submit the search

# get all section title elements
title_list = driver.find_elements_by_class_name("mw-headline")

# all expected section titles specific to this article
expected_titles = ["History", "Headquarters and offices",
    "Products and services", "Research",
    "Brand and reputation", "People and culture",
    "Employees", "IBM alumni", "Board and shareholders"]

# checking that titles are what we expect
for title, expected_title in zip(title_list, expected_titles):
    assert title.text == expected_title

driver.close()                                          # close the webdriver

We will now go back and explain each section in more detail to understand what’s going on.

Instantiating a WebDriver

driver = webdriver.Firefox()

The first thing we need to do is create an instance of the WebDriver to use throughout our script and control the browser. As mentioned in the setup section, there are multiple different driver implementations corresponding to the many popular browsers.

In this section of our script, we use the Firefox driver, but the syntax is very similar for the alternatives. More details can be found here.

Navigating to a Webpage

driver.get("https://www.wikipedia.org")

Once we have a driver, we must actually open the desired page in the browser. This can be done with the simple line above, which takes us to the Wikipedia homepage.

Locating Elements on a Page

Selenium offers many commands to locate elements, each of which is a combination of two factors: what we use to locate elements and whether or not we want to get more than one.

We can search for elements on a page using the following attributes:

  • ID
  • Class name
  • Tag name
  • Name
  • Link text
  • Partial link text
  • CSS
  • XPath

Looking at the first example of this in our code, we see that the method clearly uses an ID as the search query, which in this case is “searchInput”.

elem = driver.find_element_by_id("searchInput")

The next factor to locating elements is how many we would like to return. In the line of code above, we return a single element, which can be seen by the use of the singular “find element” in the method call. In this next line, we instead get a list of matching elements, this time searching by class name. Note the plural form of the method:

title_list = driver.find_elements_by_class_name("mw-headline")

It’s important to note that if a singular locator method is used but multiple elements on the page would be returned, the method will return the first match. More syntax details can be found here.

Interacting with Elements

Once we have a reference to an element on a page, we most likely want to perform some actions with it. Let’s look at some cases of this in the example: 

elem = driver.find_element_by_id("searchInput")         # find the search box
elem.clear()                                            # clear the search box

elem.send_keys("IBM")                                   # input "IBM" into the search box

elem.send_keys(Keys.RETURN)                             # submit the search

The “elem” variable is a reference to the search box on the Wikipedia homepage.

We first clear the search box on the second line to prepare for our input. The last two lines are where we then apply our own input using the send_keys() method to interact with text boxes. We first straightforwardly enter the search query “IBM”, and then simulate pressing the enter key to submit the search using an enum in Selenium’s Keys class. After this line, the browser should be at IBM’s Wikipedia page.

Asserting Values on a Page

At this point, we’re a little over halfway through the example code, shown again here for reference: 

from selenium import webdriver                          # import Python libraries
from selenium.webdriver.common.keys import Keys

driver = webdriver.Firefox()                            # open webdriver for Firefox

driver.implicitly_wait(10)                              # set max wait period for driver

driver.get("https://www.wikipedia.org")                 # navigate to Wikipedia

elem = driver.find_element_by_id("searchInput")         # find the search box
elem.clear()                                            # clear the search box

elem.send_keys("IBM")                                   # input "IBM" into the search box

elem.send_keys(Keys.RETURN)                             # submit the search

# get all section title elements
title_list = driver.find_elements_by_class_name("mw-headline")

# all expected section titles specific to this article
expected_titles = ["History", "Headquarters and offices",
    "Products and services", "Research",
    "Brand and reputation", "People and culture",
    "Employees", "IBM alumni", "Board and shareholders"]

# checking that titles are what we expect
for title, expected_title in zip(title_list, expected_titles):
    assert title.text == expected_title

driver.close()                                          # close the webdriver

We’ve explained that the line initializing title_list gets a list of all elements on a page with the class name “mw-headline”. In the context of our example, we are grabbing a list of all of the section titles on IBM’s Wikipedia page. We will now use those elements to do some testing.

For this example, we’ll perform a simple test to make sure that the section titles specific to this page are written properly. This code snippet is below. 

# checking that titles are what we expect
for title, expected_title in zip(title_list, expected_titles):
    assert title.text == expected_title

At the top, we create a list of our expected titles. The for loop then loops through the expected and actual lists simultaneously, asserting that the text is equal. If the code runs with no errors, the titles are correct!

It’s important to note a couple things about the assert statement. The first is that this is the built-in assert statement for Python. Selenium is often used for UI testing, but it does not come with testing methods, leaving the tester to incorporate any testing framework they desire. Second is that the text is extracted from the title elements by using the “.text” syntax. Other properties of web elements that can be extracted can be found here.

Aside: Waits

driver.implicitly_wait(10)

You may have noticed that we skipped over a lie of the full code example, shown above. This line implements a wait, which is an important concept in Selenium. In short, waits are used to allow elements on the page to load completely before we access them, so that no unexpected behavior occurs. Selenium has two types of waits: implicit and explicit.

Implicit waits are simpler to implement, and it’s what we use in the example. To use this, we set an implicit waiting time for the driver after instantiation using the implicitly_wait() method. The one argument is a number of seconds. For the life of the driver, or until this is overwritten, all actions performed on the page will wait up to the specified amount of time for the page to load. If the page takes too long to load, a timeout exception will be raised.

Explicit waits are slightly more complicated to implement, but they offer a bit more in terms of flexibility and allow us to wait only when we want to. To demonstrate, here is the first element locator of the full example rewritten to use an explicit wait, along with the new required imports: 

from selenium import webdriver
from selenium.webdriver.common.by import By
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

driver = webdriver.Firefox()

        .
        .        # skipping intermediate code
        .

wait = WebDriverWait(driver, 10)
title_list = wait.until(EC.presence_of_element_located((By.ID, "searchInput")))

In this implementation, we specify a WebDriverWait object, which sets the maximum waiting time for the action to be performed. Inside the until clause, we put the expected condition (EC) that will break out of the waiting loop. There are multiple expected conditions defined by Selenium, which can be found here. The expected condition takes a locator as an argument, which in this case is a tuple equivalent to finding an element with the ID of “searchInput”.

More details about waits can be found here.

Small Side Idea: Scraping Dynamic Webistes

The use of waits come in very handy for many reasons, but one of these specifically is you can also use them to scrape dynamic web sites. If you just use a get on a dynamic website then read in the page source, a lot of the tags probably won't be loaded from javascript yet. In order to get around this, as mentioned above, you wait for an element to be present then grab it. This is a very easy and straightforward way as opposed to your other option which is trying to reverse engineer javascript code to figure out how it's loading the content.

Unit Testing

Selenium can easily be used for unit testing web applications. The following python script executes three unit test cases for a web application that analyzes QR codes: 

# Example Selenium test script for QR Scanner (https://webqr.com)
# The first two test cases should pass but the last should fail

from selenium import webdriver                                            # import Python libraries
from selenium.webdriver.common.keys import Keys
import unittest

class UnitTestQRScanner(unittest.TestCase):
    def setUp(self):
        self.driver = webdriver.Firefox()                                 # open webdriver for Firefox
        self.driver._is_remote = False                                    # fix for bug in webdriver
        self.driver.implicitly_wait(10)                                   # set max wait period for driver

    def test_QRScannerGoogle(self):
        driver = self.driver
        driver.get("https://webqr.com")                                   # navigate to QR Code scanner

        elem = driver.find_element_by_id("qrimg")                         # find upload button
        elem.click()                                                      # click upload button

        qr_img_directory_1 = r"C:\PATH\TO\qrgoogle.png"                   # set image upload directory
        elem_upload = driver.find_element_by_css_selector("input[type='file']")   # find qr upload
        elem_upload.send_keys(qr_img_directory_1)                         # send image to upload section

        expected_response = "https://google.com/"
        response = driver.find_element_by_xpath("//div[@id='result']//b") # get the response address
        self.assertTrue(response.text == expected_response)               # assert

    def test_QRScannerLinkedIn(self):
        driver = self.driver
        driver.get("https://webqr.com")                                   # navigate to QR Code scanner

        elem = driver.find_element_by_id("qrimg")                         # find upload button
        elem.click()                                                      # click upload button

        qr_img_directory_1 = r"C:\PATH\TO\qrlinkedin.png"                 # set image upload directory
        elem_upload = driver.find_element_by_css_selector("input[type='file']")   # find qr upload
        elem_upload.send_keys(qr_img_directory_1)                         # send image to upload section

        expected_response = "https://linkedin.com/feed"
        response = driver.find_element_by_xpath("//div[@id='result']//b") # get the response address
        self.assertEqual(response.text, expected_response)                # assert

    def test_QRScannerHuffingtonPost(self):                               # THIS TEST CASE SHOULD FAIL
        driver = self.driver
        driver.get("https://webqr.com")                                   # navigate to QR Code scanner

        elem = driver.find_element_by_id("qrimg")                         # find upload button 
        elem.click()                                                      # click upload button

        qr_img_directory_1 = r"C:\PATH\TO\qrnewyorktimes.png"             # set image upload directory
        elem_upload = driver.find_element_by_css_selector("input[type='file']")   # find qr upload
        elem_upload.send_keys(qr_img_directory_1)                         # send image to upload section

        expected_response = "http://www.huffingtonpost.com/"
        response = driver.find_element_by_xpath("//div[@id='result']//b") # get the response address
        self.assertEqual(response.text, expected_response)                # assert

    def tearDown(self):
        self.driver.quit()

if __name__ == "__main__":
    unittest.main()                                                       # start testing QR scanner

This script subclasses unittest.TestCase which keeps track of the results of the three test cases. Subclassing TestCase allows each test case to be run independently so that if any errors occur, the execution of other cases will be unaffected. TestCase does this by calling setUp() and tearDown() for each case. Also, TestCase provides thorough documentation for test cases that fail.

The first two test cases in this script test that the QR Scanner web application properly scans the QR codes and outputs the proper web address for each. Other Use Cases

You can imagine all of the different scenarios where Selenium would be useful to test your UI. You may have a Web Application with a login and registration page and want to make sure users can successfully create accounts and login to them. Maybe you want to test access to specific areas of a web page where a non premium user tries to access a section that is only available to premium users. Another use could be to test different form inputs to make sure validation is working correctly. Have Selenium give the form invalid text, then make sure it returns the same form back to you thus ensuring form validation is working correctly. Selenium can be used for a number of use cases and these are just a fraction of what can be automated.
